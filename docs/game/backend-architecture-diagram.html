03 — Data Flow
Institution Setup one-time · school name, logo, term structure
Academic Year + Term Opened receipt counter initialized at 0001
Classes Created P.1–P.7 · S.1–S.6 · tertiary
Students Enrolled linked to class · ID assigned
Fee Structures Defined per class · per term · per category
Payment Recorded by Bursar receipt number generated · channel selected · allocations created
Cashbook Entry Auto-Created never manual · always automatic
Bank Deposit Recorded cashbook entry auto-created
Expense Initiated → Approved → Paid voucher printed · cashbook entry auto-created
Reports Generated daily · term · student statement · income vs expenditure
Every Step → Audit Log who · what · when · old value · new value · append-only

05 — Rules You Must Never Break in Code
Receipt numbers are sacredNever reused. Never editable. Generated atomically per term. Format: YYYY/T#/NNNN.
Cashbook is automaticNever manually created, updated, or deleted. Every entry is the automatic result of another action.
Audit log is append-onlyNo service ever updates or deletes an audit entry. Ever. Under any circumstance.
Voided payments stay visibleVoided payments remain in the database and appear as reversal entries in the cashbook.
Frozen terms are lockedNo payments or expenses can be recorded against a closed or frozen term.
Students are never hard-deletedStatus changes to inactive. History is permanent. The record stays forever.
Passwords are always hashedbcrypt only. Never plain text. Never logged. Never returned in any API response.
Every action is auditedIf it changes data, it writes to the audit log. No exceptions. Not even minor updates.
